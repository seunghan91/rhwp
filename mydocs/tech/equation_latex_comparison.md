## 1. 배경 및 목적

한컴의 수식 입력 방식은 1990년대 HWP 초기 버전부터 자체 문법을 사용해왔다. 이 문법은 한국 워드프로세서 환경에서 독자적으로 발전했으나, 국제적으로 사실상 표준인 LaTeX 수식 문법과는 근본적으로 다르다.

한컴 디벨로퍼 포럼([LaTeX 수식 쓰레드](https://forum.developer.hancom.com/t/latex/1810))에서 사용자들이 LaTeX 지원을 강력히 요청하고 있으며, 한컴 측도 내부적으로 준비 중이라 답변한 상태이다.

rhwp는 기존 한컴 수식 호환을 유지하면서, 편집기에서 LaTeX 방식도 지원하여 수학/과학 분야 사용자들에게 차별화된 가치를 제공하는 것을 목표로 한다.

**이 문서의 목적**: 두 방식의 체계적 비교를 통해 rhwp 수식 입력 개선 방향을 수립한다.

## 2. GUI 수식 입력 방식 비교

수식을 텍스트로 직접 타이핑하는 방식은 한컴이나 LaTeX나 크게 다르지 않다. 차이가 두드러지는 것은 **마우스와 아이콘을 사용한 GUI 기반 수식 입력 방식**이다. 이 절에서는 한컴 수식편집기, MS Word 수식 도구, MathType, LyX, 그리고 현재 rhwp 구현을 비교한다.

### 2.1 편집기별 GUI 입력 방식 개요

#### 한컴 수식편집기

- **입력 패러다임**: 별도 대화상자(모달) + 스크립트 직접 편집
- **도구 모음**: 상단에 수식 구조/기호 아이콘 배열. 클릭 시 커서 위치에 한컴 스크립트 코드 삽입
- **미리보기**: 스크립트 입력과 동시에 상단 미리보기 영역에 렌더링 결과 표시
- **편집 방식**: 텍스트 영역에서 스크립트를 직접 편집. 렌더링 결과를 직접 클릭/수정하는 WYSIWYG는 아님
- **특징**: 도구 버튼이 구조(분수, 제곱근 등)와 기호(그리스 문자, 연산자)로 분류. 비교적 단순한 UI

#### MS Word 수식 도구 (Equation Tools)

- **입력 패러다임**: 문서 본문 내 인라인 WYSIWYG 편집
- **삽입**: `삽입 > 수식` 메뉴 또는 `Alt+=` 단축키로 수식 영역 생성
- **리본 탭 구성**:
  - **기호(Symbols)**: 기호 갤러리 — 그리스 문자, 연산자, 화살표 등을 카테고리별로 탐색·삽입
  - **구조(Structures)**: 분수, 첨자, 제곱근, 적분, 행렬, 괄호 등 시각적 템플릿 갤러리. 클릭 시 플레이스홀더(점선 상자)가 삽입되고, 각 상자에 값을 입력
- **듀얼 모드**:
  - **Professional(전문가)**: 렌더링된 수식을 직접 편집하는 WYSIWYG 모드. 수식 구조를 시각적으로 확인하면서 편집
  - **Linear(선형)**: 텍스트 기반 입력 모드. UnicodeMath 또는 LaTeX 문법으로 수식을 타이핑하면 자동 변환
- **자동변환**: Linear 모드에서 `\alpha` 입력 후 스페이스 → α로 즉시 변환. `_` `^` 입력 시 첨자 자동 구조화
- **잉크 입력**: 터치/펜 기기에서 손글씨 → 수식 자동 인식 (`그리기 > 잉크를 수학으로 변환`)
- **특징**: 별도 대화상자 없이 문서 본문에서 직접 편집. 가장 진보된 WYSIWYG 수식 입력 경험

#### MathType

- **입력 패러다임**: 별도 윈도우 + 완전한 WYSIWYG 편집
- **도구 모음**: 상단에 9개 이상의 기호/템플릿 탭바. 각 탭에 관련 기호·구조 아이콘 배열
  - 탭1: 관계 기호, 탭2: 공백/장식, 탭3: 윗줄/아랫줄, 탭4: 연산자, 탭5: 화살표
  - 탭6: 논리/집합, 탭7: 그리스 소문자, 탭8: 그리스 대문자, 탭9: 수학 기호
  - 하단 탭: 구조 템플릿 (분수, 제곱근, 첨자, 행렬, 적분, 괄호 등)
- **편집 방식**: 렌더링된 수식을 직접 클릭하여 커서를 놓고, 키보드로 입력하면 즉시 포맷팅됨. 완전한 WYSIWYG
- **LaTeX 연동**: TeX/LaTeX 마크업 직접 입력 가능. MathType ↔ LaTeX 양방향 변환 지원
- **특징**: 가장 오래된(1987~) 전문 수식 편집기. 풍부한 기호·템플릿 라이브러리. Word/PowerPoint OLE 통합

#### LyX

- **입력 패러다임**: WYSIWYM (What You See Is What You Mean) — 의미 구조 편집
- **수식 모드**: `Ctrl+M` 또는 메뉴로 수식 모드 진입. 수식 영역이 파란 프레임으로 표시
- **도구바**: 수학 패널에서 기호·구조 클릭 삽입 + 키보드 LaTeX 명령어 직접 입력 가능
- **편집 방식**: 수식을 입력하면 즉시 근사 렌더링이 표시되지만, 최종 결과는 LaTeX 컴파일 후에만 확인 가능. Professional WYSIWYG와 LaTeX 직접 편집의 중간 지점
- **LaTeX 연동**: 내부적으로 LaTeX 기반. `\frac`, `\sqrt` 등 LaTeX 명령어를 수식 모드에서 직접 타이핑 가능
- **특징**: 학술 문서 작성에 최적화. LaTeX의 조판 품질 + GUI 편의성의 절충안

### 2.2 GUI 입력 방식 분류 체계

수식 편집기의 GUI 입력 방식을 3가지 축으로 분류할 수 있다:

#### 축 1: 편집 위치 (문서 내 vs 별도 대화상자)

| 방식 | 편집기 | 장점 | 단점 |
|------|--------|------|------|
| **인라인 (문서 내)** | MS Word | 컨텍스트 유지, 즉각적 결과 확인 | 구현 복잡도 높음 |
| **별도 대화상자** | 한컴, MathType, rhwp | 구현 단순, 독립적 UI | 컨텍스트 전환 필요 |
| **하이브리드** | LyX | 문서 내 편집 + 별도 수학 패널 | 학습 곡선 |

#### 축 2: 렌더링 방식 (WYSIWYG vs 스크립트+미리보기)

| 방식 | 편집기 | 장점 | 단점 |
|------|--------|------|------|
| **완전한 WYSIWYG** | MS Word, MathType | 직관적, 초보자 친화적 | 복잡한 수식 편집 시 오히려 불편 |
| **스크립트+미리보기** | 한컴, rhwp | 정밀한 제어 가능, 복사/붙여넣기 용이 | 스크립트 문법 학습 필요 |
| **WYSIWYM** | LyX | 구조 파악 용이, LaTeX 품질 | 최종 결과 즉시 확인 불가 |

#### 축 3: 구조 삽입 방식 (템플릿 갤러리 vs 명령어 입력)

| 방식 | 편집기 | 장점 | 단점 |
|------|--------|------|------|
| **시각적 갤러리** | MS Word (Structures) | 구조를 시각적으로 선택, 발견성 높음 | 갤러리 탐색 시간 |
| **아이콘+텍스트 삽입** | 한컴, rhwp | 빠른 접근, 스크립트 학습 도움 | 아이콘만으로 기능 파악 어려움 |
| **풍부한 탭 도구바** | MathType | 대규모 기호 라이브러리, 분류 체계적 | 탭 전환 빈번 |
| **명령어 자동완성** | LyX, VS Code + LaTeX | 숙련자에게 최고 속도 | 명령어 암기 필요 |

### 2.4 rhwp 현재 GUI 입력 방식 분석

현재 rhwp의 수식 편집기(`equation-editor-dialog.ts`)는 한컴 방식을 기반으로 구현되어 있다:

```
┌─────────────────────────────────────────────┐
│  수식 편집                              [×]  │
├─────────────────────────────────────────────┤
│  [분수][x²][x₂][√][Σ][∫][()] | [α][β]...   │  ← 도구 모음 (48개 버튼)
│  [[][행렬][hat][bar] | [±][×][÷][≠]...      │
├─────────────────────────────────────────────┤
│  ┌─────────────────────────────────────┐    │
│  │     SVG 미리보기 영역                │    │  ← 실시간 미리보기
│  └─────────────────────────────────────┘    │
├─────────────────────────────────────────────┤
│  ┌─────────────────────────────────────┐    │
│  │ {-b PLUSMINUS SQRT{b^{2}-4ac}}     │    │  ← 스크립트 입력 (textarea)
│  │ OVER {2a}                           │    │
│  └─────────────────────────────────────┘    │
├─────────────────────────────────────────────┤
│  글자 크기 [10] pt    색 [■]                │  ← 속성 패널
├─────────────────────────────────────────────┤
│                          [확인] [취소]       │
└─────────────────────────────────────────────┘
```

**현재 방식의 특징**:
- 도구 버튼 클릭 → textarea 커서 위치에 한컴 스크립트 삽입 → `{}` 안에 커서 자동 배치
- WASM `renderEquationPreview()` 호출로 300ms 디바운스 실시간 미리보기
- 48개 템플릿: 구조 11개 + 그리스 문자 9개 + 연산자/기호 10개 = 30개 (나머지는 그룹 구분)

**현재 방식의 한계**:
1. WYSIWYG 부재 — 렌더링 결과를 직접 클릭/편집할 수 없음
2. 스크립트 전용 — 스크립트 문법을 모르면 도구 버튼에만 의존
3. 자동완성 없음 — 명령어 타이핑 보조 기능 없음
4. 템플릿 수 제한 — 48개로 한컴 수식편집기 대비 소규모
5. 기호 검색 없음 — 원하는 기호를 이름/설명으로 검색 불가

### 2.5 GUI 입력 방식 개선 방향

| 개선 영역 | 현재 | 목표 | 우선순위 | 참고 모델 |
|----------|------|------|---------|----------|
| **LaTeX 입력 모드** | 미지원 | 한컴/LaTeX 듀얼 모드 전환 | 높음 | MS Word Linear 모드 |
| **자동완성** | 없음 | 명령어 입력 시 드롭다운 추천 | 높음 | VS Code, LyX |
| **템플릿 확장** | 48개 | 100+ (카테고리별 탭 분류) | 중간 | MathType 탭 구조 |
| **기호 검색** | 없음 | 이름/유니코드로 기호 검색 | 중간 | MS Word 기호 갤러리 |
| **시각적 구조 갤러리** | 텍스트 레이블 | 미니 SVG 미리보기 아이콘 | 중간 | MS Word Structures |
| **WYSIWYG 편집** | 미지원 | 렌더링 결과 직접 편집 | 낮음 (장기) | MS Word, MathType |
| **키보드 단축키** | Ctrl+Enter만 | 구조별 단축키 (Ctrl+/, Ctrl+Shift+S 등) | 중간 | MathType |
| **Undo/Redo** | 미지원 | 스크립트 편집 이력 관리 | 중간 | 표준 |

### 2.6 핵심 통찰

1. **MS Word가 현재 가장 진보된 GUI 수식 편집 경험을 제공**한다. 인라인 WYSIWYG + Professional/Linear 듀얼 모드 + 자동변환은 모든 수준의 사용자를 만족시킨다.

2. **완전한 WYSIWYG는 장기 목표**로 두되, 단기적으로는 **듀얼 입력 모드(한컴/LaTeX) + 자동완성**이 가장 효과적인 개선이다. 이는 기존 스크립트+미리보기 구조를 유지하면서도 사용성을 크게 향상시킨다.

3. **MathType의 탭 기반 도구바 구조**는 기호·템플릿 수를 대폭 확장하면서도 UI를 깔끔하게 유지하는 좋은 참고 모델이다.

4. **rhwp의 차별화 전략**: 한컴 호환 + LaTeX 지원 + 웹 네이티브(WASM). 이 조합을 제공하는 수식 편집기는 현재 존재하지 않는다.

## 3. 명령어 대응표

### 3.1 구조 명령어

| 기능 | 한컴 | LaTeX | 구조적 차이 |
|------|------|-------|------------|
| 분수 | `a OVER b` | `\frac{a}{b}` | 한컴: 중위 연산자, LaTeX: 전위 함수 |
| 분수 (선 없음) | `a ATOP b` | `{a \atop b}` 또는 `\genfrac{}{}{}{}` | 유사 |
| 제곱근 | `SQRT{x}` | `\sqrt{x}` | 거의 동일 |
| n제곱근 | `ROOT{n}{x}` | `\sqrt[n]{x}` | 한컴: 중괄호 2개, LaTeX: 대괄호+중괄호 |
| 위첨자 | `x^{2}` | `x^{2}` | **동일** |
| 아래첨자 | `x_{n}` | `x_{n}` | **동일** |
| 조합 | `BINOM{n}{r}` | `\binom{n}{r}` | 거의 동일 |
| 색상 | `COLOR{255,0,0}{text}` | `\textcolor{red}{text}` | 한컴: RGB 직접, LaTeX: 색상명/모델 |

### 3.2 큰 연산자

| 기능 | 한컴 | LaTeX | 차이 |
|------|------|-------|------|
| 적분 | `INT_{a}^{b}` | `\int_{a}^{b}` | `\` 접두어 유무 |
| 이중적분 | `DINT` | `\iint` | 명령어명 차이 |

| 기능 | 한컴 | LaTeX | 차이 |
|------|------|-------|------|
| 행렬 (괄호 없음) | `MATRIX{a#b;c#d}` | `\begin{matrix}a&b\\c&d\end{matrix}` | **구조적 차이 매우 큼** |
| 소괄호 행렬 | `PMATRIX{a#b;c#d}` | `\begin{pmatrix}a&b\\c&d\end{pmatrix}` | 동일 |
| 대괄호 행렬 | `BMATRIX{a#b;c#d}` | `\begin{bmatrix}a&b\\c&d\end{bmatrix}` | 동일 |

## 4. 문법 구조 비교

### 4.1 설계 철학

| 관점 | 한컴 방식 | LaTeX 방식 |
|------|----------|-----------|
| **명령어 접두어** | 없음 (키워드 그대로) | `\` 백슬래시 필수 |
| **구조 표현** | 중위 연산자 중심 (`a OVER b`) | 전위 함수 중심 (`\frac{a}{b}`) |
| **환경 블록** | 없음 (`MATRIX{...}`) | `\begin{env}...\end{env}` |
| **행 구분** | `#` (줄바꿈), `;` (행렬 행) | `\\` (공통) |
| **열 구분** | `#` (행렬), `&` (정렬) | `&` (공통) |
| **대소문자** | 대소문자 무시 (OVER=over=Over) | 대소문자 구분 (\Alpha ≠ \alpha) |
| **문법 일관성** | 명령어마다 다른 패턴 | `\명령어{인자}` 통일 패턴 |

### 4.2 동일한 수식의 비교 예시

**예시 1: 이차방정식의 근의 공식**

```
# 한컴
x = {-b PLUSMINUS SQRT{b^{2} - 4ac}} OVER {2a}

# LaTeX
x = \frac{-b \pm \sqrt{b^{2} - 4ac}}{2a}
```

**예시 2: 정적분**

```
# 한컴
INT_{0}^{1} x^{2} dx = {1} OVER {3}

# LaTeX
\int_{0}^{1} x^{2} \, dx = \frac{1}{3}
```

**예시 3: 행렬**

```
# 한컴
PMATRIX{a#b;c#d}

# LaTeX
\begin{pmatrix} a & b \\ c & d \end{pmatrix}
```

**예시 4: 연립방정식**

```
# 한컴
CASES{x + y = 1 # x - y = 3}

# LaTeX
\begin{cases} x + y = 1 \\ x - y = 3 \end{cases}
```

### 4.3 핵심 구조적 차이 요약

1. **`\` 접두어**: LaTeX는 모든 명령어에 백슬래시를 사용하여 명령어와 변수를 명확히 구분한다. 한컴은 키워드 자체를 명령어로 인식하여 변수명과 충돌 가능성이 있다.
2. **중위 vs 전위**: 한컴의 `a OVER b`는 직관적이지만, LaTeX의 `\frac{a}{b}`는 인자의 범위가 중괄호로 명확하다.
3. **환경 블록**: LaTeX의 `\begin{...}\end{...}` 방식은 복잡한 구조(다단 정렬, 여러 행 수식)에서 강력하다. 한컴은 단일 중괄호로 모든 것을 처리한다.
4. **행/열 구분자 혼용**: 한컴은 `#`이 줄바꿈과 열 구분을 겸하고, `;`이 행렬 행 구분이다. LaTeX는 `\\`(행), `&`(열)로 일관적이다.

## 5. 장단점 비교 분석

### 5.1 한컴 방식

| 장점 | 단점 |
|------|------|
| 한국 사용자 친숙 (30년+ 역사) | 국제적으로 거의 미사용 |
| 기존 HWP 문서 완벽 호환 | 학술 커뮤니티에서 비표준 |
| 간단한 수식은 직관적 (`a OVER b`) | 복잡한 수식 표현 시 가독성 저하 |
| 학습 부담 낮음 (한국어 환경) | 온라인 참고자료/예제 부족 |
| 대소문자 무시로 입력 편리 | 글꼴 스타일 확장 제한적 (mathbb, mathcal 없음) |
| | 행/열 구분자가 혼란스러움 (#, ;, &) |
| | 명령어와 변수명 충돌 가능 |

### 5.2 LaTeX 방식

| 장점 | 단점 |
|------|------|
| 국제 표준 (학술/과학/공학 보편) | 학습 곡선 높음 (`\` 접두어, 환경 블록 문법) |
| 방대한 온라인 자료/튜토리얼 | 간단한 수식도 `\` 접두어 필수 |
| 풍부한 도구 생태계 (MathJax, KaTeX, Overleaf) | `\begin{...}\end{...}` 타이핑 부담 |
| 문법 일관성 (`\명령어{인자}`) | HWP 문서 호환을 위한 변환 필요 |
| `\mathbb`, `\mathcal` 등 확장 글꼴 지원 | 대소문자 구분 엄격 |
| 명령어-변수 구분 명확 (`\alpha` vs `alpha`) | |
| 복사/붙여넣기로 타 시스템 호환 | |
| AI/LLM이 LaTeX 수식 생성 능력 우수 | |

## 7. 현재 파서 아키텍처 분석

### 7.1 파이프라인 구조

```
수식 스크립트 (문자열)
    ↓
tokenizer.rs: Tokenizer → Vec<Token>
    - TokenType 13종: Command, Number, Symbol, Text, LBrace, RBrace,
      LParen, RParen, LBracket, RBracket, Subscript, Superscript,
      Whitespace, Quoted, Eof
    ↓
parser.rs: EqParser (재귀 하강) → EqNode (AST)
    - 25종 노드: Row, Text, Number, Symbol, MathSymbol, Function,
      Fraction, Atop, Sqrt, Superscript, Subscript, SubSup, BigOp,
      Limit, Matrix, Cases, Pile, EqAlign, Rel, Paren, Decoration,
      FontStyle, Color, Space, Newline, Quoted, Empty
    ↓
layout.rs: EqLayout → LayoutBox (위치/크기/기준선)
    ↓
svg_render.rs / canvas_render.rs → SVG 조각 / Canvas 2D API
```

### 7.2 핵심 설계 특성

1. **토크나이저**: 문자 단위 순회, 명령어는 영문자 연속으로 추출. `\` 처리 로직 없음.
2. **파서**: `parse_command()` 메서드에서 명령어별 분기. 대소문자 무시 비교(`cmd_eq`).
3. **AST**: 한컴/LaTeX 독립적인 범용 구조. `EqNode::Fraction`, `EqNode::Sqrt` 등은 양쪽 모두에서 생성 가능.
4. **레이아웃/렌더러**: AST(EqNode)만 입력으로 받음. 입력 문법에 무관.

### 7.3 LaTeX 문법 수용 시 영향 분석

| 레이어 | 변경 필요성 | 설명 |
|--------|-----------|------|
| **AST (ast.rs)** | **변경 불필요** | EqNode는 이미 문법 독립적. \frac도 Fraction 노드로 변환 가능 |
| **레이아웃 (layout.rs)** | **변경 불필요** | LayoutBox는 AST만 입력으로 받음 |
| **SVG 렌더러 (svg_render.rs)** | **변경 불필요** | LayoutBox만 입력으로 받음 |
| **Canvas 렌더러 (canvas_render.rs)** | **변경 불필요** | 동일 |
| **토크나이저 (tokenizer.rs)** | **변경 필요** | `\` 접두어 처리, `\\` 줄바꿈 등 |
| **파서 (parser.rs)** | **변경 필요** | `\begin{env}...\end{env}` 환경 블록, `\frac{a}{b}` 전위 구문 등 |
| **기호 맵 (symbols.rs)** | **확장 필요** | `\mathbb`, `\mathcal` 등 추가 |

**핵심 발견**: AST 이하 레이어(layout, render)는 완전히 문법 독립적이므로, **토크나이저와 파서만 확장하면 LaTeX 지원이 가능**하다. 이는 현재 아키텍처의 큰 강점이다.

## 8. 확장 전략 비교

### 전략 A: 듀얼 토크나이저 방식

```
입력 문자열
    ↓
[자동 감지] → '\' 발견 시 LaTeX 모드, 아니면 한컴 모드
    ↓                    ↓
LaTeX 토크나이저    한컴 토크나이저 (기존)
    ↓                    ↓
LaTeX 파서          한컴 파서 (기존)
    ↓                    ↓
    └─── 공통 AST (EqNode) ───┘
              ↓
         layout → render
```

| 항목 | 평가 |
|------|------|
| **구현 난이도** | 중간 — 새 토크나이저+파서 작성, 기존 코드 수정 최소 |
| **유지보수** | 양호 — 두 경로 독립적, 기존 코드 영향 없음 |
| **혼합 입력** | 불가 — 한 수식 내에서 한컴+LaTeX 혼용 불가 |
| **위험도** | 낮음 — 기존 한컴 파서에 영향 없음 |

### 전략 B: 전처리 변환기 방식

```
LaTeX 입력 문자열
    ↓
[LaTeX→한컴 변환기] — \frac{a}{b} → {a} OVER {b}
    ↓
한컴 토크나이저 (기존)
    ↓
한컴 파서 (기존)
    ↓
AST → layout → render
```

| 항목 | 평가 |
|------|------|
| **구현 난이도** | 낮음 — 텍스트 치환/정규식 기반 변환기 |
| **유지보수** | 취약 — 변환 규칙이 복잡해질수록 버그 증가 |
| **표현 범위** | 제한적 — 한컴 문법에 대응 없는 LaTeX 기능 변환 불가 (mathbb 등) |
| **위험도** | 중간 — 변환 손실 발생 가능 |

### 전략 C: 통합 파서 방식

```
입력 문자열
    ↓
통합 토크나이저 ('\' 감지 시 LaTeX 토큰 발행)
    ↓
통합 파서 (한컴 명령어 + LaTeX 명령어 모두 처리)
    ↓
공통 AST (EqNode)
    ↓
layout → render
```

| 항목 | 평가 |
|------|------|
| **구현 난이도** | 높음 — 기존 토크나이저/파서 대폭 수정 |
| **유지보수** | 복잡 — 두 문법이 하나의 코드에 얽힘 |
| **혼합 입력** | 가능 — 한 수식 내에서 한컴+LaTeX 혼용 가능 |
| **위험도** | 높음 — 기존 한컴 파서 동작에 회귀 위험 |

### 전략 비교 요약

| 기준 | A: 듀얼 | B: 전처리 | C: 통합 |
|------|---------|----------|---------|
| 구현 난이도 | ★★★ | ★★ | ★★★★ |
| 기존 코드 보호 | ★★★★★ | ★★★★ | ★★ |
| 표현 범위 | ★★★★★ | ★★★ | ★★★★★ |
| 혼합 입력 | ★ | ★ | ★★★★★ |
| 유지보수성 | ★★★★ | ★★ | ★★ |
| 확장성 | ★★★★ | ★★ | ★★★ |

## 9. 개선 방향 제안

### 9.1 권장 전략: A (듀얼 토크나이저) + 향후 C 확장

**1차 구현 (별도 타스크)**: 전략 A — 듀얼 토크나이저 방식

**선정 근거**:
1. **기존 코드 무수정**: 한컴 파서(tokenizer.rs, parser.rs)를 전혀 건드리지 않으므로 기존 HWP 문서 호환에 영향 없음
2. **완전한 LaTeX 표현 범위**: 전처리 변환(B)과 달리 한컴 문법에 없는 `\mathbb`, `\mathcal` 등도 AST에 새 노드로 추가하여 지원 가능
3. **명확한 분리**: 한컴/LaTeX 파서가 독립적이므로 각각 테스트·유지보수 가능
4. **점진적 확장 가능**: 향후 혼합 입력이 필요해지면 통합 파서(C)로 전환 가능

**장기 확장**: 사용자 피드백에 따라 혼합 입력 지원이 필요하면 전략 C로 전환

### 9.2 구현 범위 (별도 타스크 분리 계획)

이번 타스크(#139)는 **분석 및 방향 수립**까지만 수행한다. 실제 LaTeX 파서 구현은 별도 타스크로 분리한다.

**향후 타스크에서 구현할 항목**:

1. **LaTeX 토크나이저** (`src/renderer/equation/latex_tokenizer.rs`)
   - `\` 접두어 명령어 처리
   - `\\` 줄바꿈, `&` 열 구분
   - `\begin{env}...\end{env}` 환경 블록 토큰화

2. **LaTeX 파서** (`src/renderer/equation/latex_parser.rs`)
   - `\frac{a}{b}` 전위 구문 → EqNode::Fraction
   - `\begin{pmatrix}...\end{pmatrix}` → EqNode::Matrix
   - `\begin{cases}...\end{cases}` → EqNode::Cases
   - `\mathbb{R}` 등 확장 글꼴 → EqNode::FontStyle (새 스타일 추가)

3. **모드 전환 로직** (`src/renderer/equation/mod.rs`)
   - **UI 명시적 모드 전환을 1순위로 채택** (피드백 반영: `mydocs/feedback/task139-eq-01.md`)
   - 자동 감지(첫 비공백 `\`)는 보조 수단으로만 사용
   - 이유: 사용자가 `x = \frac{1}{2}` 처럼 변수부터 입력하면 첫 글자 감지가 실패. 중간에 `\` 발견 시 "LaTeX 모드로 전환하시겠습니까?" 힌트를 제공하는 UX가 필요

4. **수식 편집 UI 확장** (`equation-editor-dialog.ts`)
   - **입력 모드 토글 버튼** (한컴 ↔ LaTeX) — 현재 파싱 모드를 명시적으로 표시
   - 한컴 모드 파싱 중 `\` 발견 시 LaTeX 전환 힌트 제공
   - LaTeX 전용 템플릿 버튼 추가 (모드별 자동 전환)
   - LaTeX 자동완성/도움말

5. **AST 확장** (`ast.rs`)
   - `FontStyleKind` 확장: `Blackboard`, `Calligraphy`, `Fraktur` 추가
   - 필요 시 `Overbrace`, `Underbrace`, `Phantom` 등 새 노드

## 10. 결론

1. **한컴 수식 호환은 반드시 유지**한다 — 기존 HWP 문서의 수식을 완벽하게 파싱·렌더링해야 한다.
2. **LaTeX 입력 지원은 전략적으로 필수**이다 — 학술/과학 분야 사용자 확보와 AI 시대의 수식 호환에 핵심적이다.
3. **듀얼 토크나이저 방식(전략 A)을 권장**한다 — 기존 코드 보호, 완전한 표현 범위, 독립적 유지보수의 균형이 가장 우수하다.
4. **현재 AST 아키텍처가 큰 강점**이다 — 토크나이저/파서만 확장하면 layout/render는 무수정으로 LaTeX를 지원할 수 있다.
5. **실제 LaTeX 파서 구현은 별도 타스크**로 분리한다 — 이번 타스크에서는 분석과 방향 수립까지만 수행한다.
