## 2. TeX/LaTeX 표준 레이아웃 파라미터

TeX의 수식 레이아웃은 Knuth의 *The TeXbook*에 명시된 파라미터를 기반으로 한다. 한컴 수식 편집기가 TeX를 참고하여 개발되었으므로, TeX 표준값이 한컴 기준값의 근사치 역할을 한다.

### 2.1 TeX 수식 스타일별 크기

| 스타일 | 크기 비율 | 용도 |
|--------|----------|------|
| Display (D) | 1.0 | 독립 수식 |
| Text (T) | 1.0 | 인라인 수식 |
| Script (S) | 0.7 | 첨자 |
| ScriptScript (SS) | 0.5 | 이중첨자 |

**비교**: rhwp의 `SCRIPT_SCALE=0.7`은 TeX의 Script 스타일 비율과 **정확히 일치**한다.

### 2.2 TeX 분수 관련 파라미터

| 파라미터 | TeX 값 (대략) | 설명 |
|----------|-------------|------|
| default_rule_thickness | 0.04em | 분수선 두께 |
| num1 (D 스타일 분자 shift) | 0.676em | 분자 baseline → 분수선 위 거리 |
| num2 (T 스타일 분자 shift) | 0.394em | 인라인 분자 shift |
| denom1 (D 스타일 분모 shift) | 0.686em | 분모 baseline → 분수선 아래 거리 |
| denom2 (T 스타일 분모 shift) | 0.345em | 인라인 분모 shift |
| axis_height | 0.25em | 수식 축 높이 (분수선 위치 기준) |

**비교**: rhwp의 `FRAC_LINE_THICK=0.04`는 TeX의 `default_rule_thickness`와 **일치**. `FRAC_LINE_PAD=0.15`는 TeX의 num/denom shift 방식과 다른 접근(균일 여백).

### 2.3 TeX 첨자 관련 파라미터

| 파라미터 | TeX 값 (대략) | 설명 |
|----------|-------------|------|
| sup1 (위첨자 shift) | 0.413em | D 스타일 위첨자 올림 |
| sup2 | 0.363em | T 스타일 위첨자 올림 |
| sub1 (아래첨자 shift) | 0.15em | 아래첨자 내림 |
| sub_drop | 0.05em | 아래첨자 추가 내림 |
| sup_drop | 0.386em | 위첨자 올림 기준 |

**비교**: rhwp의 위첨자 `baseline - sup_h * 0.7`, 아래첨자 `baseline * 0.4` 방식은 TeX와 다른 접근이지만, 시각적으로는 유사한 결과를 낸다.

### 2.4 TeX 큰 연산자 관련

| 파라미터 | TeX 값 (대략) | 설명 |
|----------|-------------|------|
| big_op_spacing1 | 0.111em | 위 한계 위 간격 |
| big_op_spacing2 | 0.167em | 위 한계 아래 간격 |
| big_op_spacing3 | 0.2em | 아래 한계 위 간격 |
| big_op_spacing4 | 0.6em | 위 한계-기호 최소 간격 |
| big_op_spacing5 | 0.1em | 전체 위/아래 여백 |
| Display 크기 배율 | ~1.2 | 큰 연산자 확대 비율 |

**비교**: rhwp의 `BIG_OP_SCALE=1.5`는 TeX 표준(~1.2)보다 **25% 더 크다**. 한컴이 큰 연산자를 더 크게 렌더링하는지 확인 필요.

## 4. 구조별 레이아웃 분석 및 보정 방향

### 4.1 분수 (Fraction)

**현재 방식**: 분자/분모에 균일한 `FRAC_LINE_PAD` 여백 적용
**TeX 방식**: num_shift/denom_shift로 baseline 기준 위치 지정

**보정 방향**:
- 현재 방식이 시각적으로 충분한지 한컴 비교로 확인
- 분수선 위치가 수식 축(axis_height, ~0.25em) 위치와 일치하는지 검증
- 분자/분모 간 여백이 한컴과 일치하는지 측정

### 4.2 행렬 (Matrix)

**현재 방식**: `MATRIX_COL_GAP=0.8`, `MATRIX_ROW_GAP=0.3` 고정 비율
**한컴 추정**: 행렬 셀 크기에 따라 가변적일 가능성

**보정 방향**:
- 한컴에서 2×2, 3×3, 4×4 행렬 렌더링 → 열/행 간격 측정
- 셀 내용이 길 때 간격 변화 여부 확인
- 괄호 폭(현재 `fs*0.3`)과 여백(현재 `fs*0.2`) 검증

### 4.3 큰 연산자 (BigOp)

**현재 방식**: `BIG_OP_SCALE=1.5`
**TeX 표준**: ~1.2

**보정 방향**:
- 한컴 편집기에서 ∫, Σ, Π 의 기호 크기 측정
- 첨자와 기호 사이 간격(현재 `fs*0.05`) 검증
- baseline 위치(현재 `op_h*0.6`) 검증

### 4.4 첨자 (Script)

**현재 방식**: 위첨자 `baseline - sup_h * 0.7`, 아래첨자 `baseline * 0.4`
**TeX 방식**: sup_shift, sub_shift 절대값

**보정 방향**:
- 한컴에서 `x^2`, `x_n`, `x_n^2` 렌더링 → 첨자 위치 측정
- 이중첨자(ScriptScript) 크기 비율 확인 (현재 0.7^2=0.49, TeX 표준 0.5)

### 4.5 괄호 (Paren)

**현재 방식**: 내부 여백 `PAREN_PAD=0.08`, 괄호 폭 `fs*0.3`

**보정 방향**:
- 한컴에서 `LEFT(`, `LEFT[`, `LEFT{` 렌더링 → 괄호 폭/여백 측정
- 내용물 높이에 따른 괄호 스케일링 검증

### 4.6 연산자 여백 (Symbol Padding)

**현재 방식**: 이항 연산자 ±15%, 기타 ±5%
**TeX 방식**: thin/medium/thick space (mu 단위)

**보정 방향**:
- 한컴에서 `a + b`, `a = b`, `a × b` 렌더링 → 기호 좌우 여백 측정
- 관계 연산자(=, <, >)와 이항 연산자(+, -, ×)의 여백 차이 확인

## 5. 문단 배치 분석

### 5.1 수식 컨트롤의 문단 내 배치

`shape_layout.rs`에서 수식 컨트롤의 배치 관련 코드:

- **독립 수식** (treat_as_char=false): `common.width/height`(HWPUNIT)를 그대로 사용하여 정렬(중앙/왼쪽/오른쪽)
- **인라인 수식** (treat_as_char=true): `paragraph_layout.rs`에서 텍스트 흐름 내 배치, baseline 정렬

### 5.2 높이/너비 계산

현재 수식의 높이/너비는 두 가지 값이 존재:
1. **HWP 파일의 `common.width/height`** — 한컴 편집기가 저장한 값 (HWPUNIT)
2. **rhwp 레이아웃 엔진이 계산한 값** — AST → LayoutBox 과정에서 산출

**문제**: 두 값이 불일치할 수 있음. 현재는 HWP의 값을 문단 배치에 사용하고, 레이아웃 엔진의 값을 수식 내부 렌더링에 사용.

**보정 방향**:
- HWP 저장값과 레이아웃 엔진 산출값의 차이를 측정
- 수식 내부 렌더링이 `common.width/height` 영역 내에 올바르게 맞춰지는지 확인
- 필요 시 LayoutBox의 크기를 `common.width/height`에 맞춰 스케일링

### 5.3 베이스라인 정렬

인라인 수식에서 수식과 주변 텍스트의 수직 정렬:

```rust
// paragraph_layout.rs
let eq_h = hwpunit_to_px(eq.common.height as i32, self.dpi);
let eq_y = (y + baseline - eq_h).max(y);
```

- 수식의 하단이 텍스트 baseline에 맞춰짐
- 수식 내부의 baseline(분수선 위치 등)과 텍스트 baseline의 정렬은 미보장

**보정 방향**:
- 한컴에서 인라인 수식 렌더링 → 베이스라인 정렬 관계 확인
- 수식 LayoutBox의 baseline 정보를 문단 배치에 활용하도록 개선

## 6. 텍스트 폭 추정 정밀화

### 6.1 현재 추정 방식

`estimate_text_width()` 함수가 문자별 고정 비율로 폭을 추정한다:

| 문자 유형 | 비율 (font_size 대비) |
|----------|---------------------|
| 대문자 (A-Z) | 0.65 |
| 소문자 (a-z), 숫자 (0-9) | 0.55 |
| 기호 | 0.5 |
| CJK / 유니코드 | 1.0 |
| 이탤릭 보정 | ×1.05 |

### 6.2 문제점

- 문자별 폭 차이가 크다: `i`(~0.3), `m`(~0.9), `W`(~0.9) 등
- 수식 전용 폰트(Latin Modern Math) 적용 후 글리프 폭이 변경될 수 있음
- 현재 추정은 **시각적으로 대략적** → 정밀한 치수 비교에 한계

### 6.3 보정 방향

**단기 (이번 타스크 범위)**: 한컴 비교를 통해 주요 글리프 폭 비율 보정

**장기 (후속 타스크)**:
- Canvas API `measureText()` 활용: 실제 렌더링 폰트의 글리프 폭 측정
- OpenType MATH 테이블의 메트릭 활용: Latin Modern Math의 글리프별 정밀 폭 정보
- SVG에서는 추정 유지, Canvas에서는 실측 사용하는 하이브리드 방식

## 7. 한컴 기준값 측정 방법

### 7.1 측정 도구

1. **한컴 편집기에서 표준 수식 생성** → PDF/이미지 내보내기
2. **이미지 뷰어에서 픽셀 측정** (화면 배율 고정)
3. **rhwp `export-svg`로 동일 수식 렌더링** → SVG 치수 비교
4. **rhwp `dump` 명령으로 수식 속성 확인** → HWPUNIT 치수

### 7.2 측정 대상 수식 목록

| 수식 | 측정 항목 | 스크립트 |
|------|----------|---------|
| 단순 분수 | 분수선 위치, 분자/분모 간격 | `1 OVER 2` |
| 중첩 분수 | 중첩 시 크기 축소 여부 | `{a OVER b} OVER {c OVER d}` |
| 위첨자 | 첨자 위치, 크기 | `x^{2}` |
| 아래첨자 | 첨자 위치, 크기 | `x_{n}` |
| 위·아래첨자 | 양쪽 첨자 배치 | `x_{n}^{2}` |
| 적분 | 기호 크기, 첨자 위치 | `INT_{0}^{1} f(x) dx` |
| 합 | 기호 크기, 첨자 위치 | `SUM_{i=1}^{n} a_{i}` |
| 2×2 행렬 | 열/행 간격, 괄호 크기 | `PMATRIX{a#b;c#d}` |
| 3×3 행렬 | 열/행 간격 스케일링 | `PMATRIX{a#b#c;d#e#f;g#h#i}` |
| 제곱근 | 근호 크기, 내부 여백 | `SQRT{x^{2}+1}` |
| 괄호 | 자동 크기, 내부 여백 | `LEFT({1 OVER 2}RIGHT)` |
| 연산자 여백 | +, =, × 좌우 간격 | `a + b = c` |

### 7.3 측정 절차

1. 한컴에서 글자 크기 10pt(기본)로 위 수식 12종 입력
2. 한컴 편집기 화면을 200% 확대 → 스크린샷 캡처
3. 이미지에서 핵심 치수 픽셀 측정 (폰트 크기 기준 비율로 환산)
4. rhwp에서 동일 수식 SVG 내보내기 → 치수 비교
5. 차이가 ≥5%인 항목 식별 → 보정 대상으로 분류

## 8. 보정 목표값 및 구현 계획

### 8.1 보정 우선순위

| 우선순위 | 항목 | 현재 값 | 보정 방향 | 근거 |
|---------|------|--------|----------|------|
| **높음** | `BIG_OP_SCALE` | 1.5 | TeX 표준 ~1.2에 가까운지 한컴 측정 | 시각적 영향 큼 |
| **높음** | `MATRIX_COL_GAP` | 0.8 | 한컴 측정으로 보정 | 행렬 가독성에 직접 영향 |
| **높음** | `MATRIX_ROW_GAP` | 0.3 | 한컴 측정으로 보정 | 행렬 가독성에 직접 영향 |
| **중간** | `FRAC_LINE_PAD` | 0.15 | 분수 분자/분모 간격 | 분수 가독성 |
